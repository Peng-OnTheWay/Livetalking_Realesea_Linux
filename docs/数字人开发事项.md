## 2026-05-26（周二）— Wav2Lip384 集成完成

### 背景

- 当前口型效果差，需要引入不蠢不蠢训练的 wav2lip256/384 模型
- 不蠢不蠢的 GitHub README 明确指出已适配 LiveTalking
- LiveTalking 2.0.2 只是小更新，不涉及模型架构变化

### 技术路线总结

| 层级 | 技术 | 说明 |
|------|------|------|
| 代码基座 | **LiveTalking 2.0.2** | 纯原始代码，不含之前优化 |
| 模型架构 | **双模型**：v2 (256) + SAM (384) | v2=6层encoder+kernel4, SAM=8层+空间注意力 |
| 权重来源 | **不蠢不蠢 V1.1 整合包** | wav2lip256.pth(215MB) + wav2lip384.pth(~2GB) |
| 推理 | PyTorch 2.11 + CUDA cu128 | GPU: RTX 5060 Ti 8GB |
| TTS | Edge TTS / 千问 DashScope | 前端可选切换 |
| 传输 | **WebRTC H.264** | :8010 端口，浏览器接收 |
| 管理端 | Flask + Bootstrap 5 | :8000 启动器，含 256/384 选择器 |

### 管线架构

```
┌─────────────────────────────────────────────────────────────────┐
│                      PHASE 1: 形象生成 (genavatar.py)              │
│                                                                   │
│  原视频 (4K mp4)                                                  │
│    │ video2imgs() → 存盘时 resize ≤1920px (max_side)              │
│    ▼                                                              │
│  full_imgs/ (1080×1920 PNG × N帧)   ← 质量保证：高清帧            │
│    │ 逐帧读入 → resize 到 720×1280 → 人脸检测 (SFD, batch=4)      │
│    │ 检测坐标 × inv_scale → 映射回 1080p 原帧                     │
│    │ 从 1080p 帧裁剪人脸 → resize 256×256                         │
│    ▼                                                              │
│  face_imgs/ (256×256 PNG × N帧)     ← 模型输入质量                │
│  coords.pkl  (y1,y2,x1,x2)           ← 贴回坐标                   │
├─────────────────────────────────────────────────────────────────┤
│                      PHASE 2: 实时推理 (wav2lip_avatar.py)         │
│                                                                   │
│  TTS 音频流 → Mel 特征 (80×16)                                    │
│  face_imgs/ 预加载到内存 (240帧 × 256×256 × 3)                    │
│    │ 按 batch_size=8 送入模型                                      │
│    ▼                                                              │
│  模型输出: 256×256 含口型人脸                                      │
│    │ paste_back_frame() → 贴回 1080p full_imgs                    │
│    ▼                                                              │
│  combine_frame (1080×1920)    ← resize 安全网：>1080p 则缩小      │
├─────────────────────────────────────────────────────────────────┤
│                      PHASE 3: 推流输出 (base_avatar.py → webrtc)   │
│                                                                   │
│  1080p BGR array                                                  │
│    │ VideoFrame.from_ndarray()                                    │
│    ▼                                                              │
│  WebRTC VideoStreamTrack                                          │
│    │ H.264 编码 (libx264)                                         │
│    ▼                                                              │
│  浏览器 <video> 播放                                              │
└─────────────────────────────────────────────────────────────────┘
```

### 已解决问题记录

| 问题 | 现象 | 根因 | 解决方案 |
|------|------|------|----------|
| **v2 warmup 崩溃** | `kernel_size=4 > input_size=3` | modelres=192 经过8层stride-2只剩3×3 | app.py区分256/384：256→256, 384→192 |
| **4K 人脸检测 OOM** | `CUDA out of memory` 2160×3840帧 | SFD检测器无法处理大图 | 检测时缩放到720×1280缩略图, 坐标映射回原帧 |
| **4K 帧内存溢出** | `cv2.OutOfMemoryError` 240帧×25MB=6GB RAM | 一次性加载所有4K帧 | 逐帧惰性读取, 用完即删(del), 两遍扫描 |
| **训练卡死** | batch_size=16 降级到8→4→2→1 极慢 | SFD 8GB显存不够 | `--face_det_batch_size 4` 直接指定 |
| **WebRTC 无画面** | `avcodec_open2(libx264)` 编码器崩溃 | 4K帧塞给H.264实时编码不兼容 | full_imgs预处理≤1920px + base_avatar运行时安全网 |
| **画面模糊** | 全图360p级模糊 | 4K帧→H.264降分辨率编码 | 统一输出≤1080p，H.264稳定高质量编码 |
| **声音卡顿** | 播放断断续续 | x264编码崩溃影响整个WebRTC通道 | 同上：修复编码问题后恢复正常 |

### 当前参数配置（RTX 5060 Ti 8GB）

| 参数 | 值 | 说明 |
|------|-----|------|
| `--batch_size` | **8** (256) / **≤4** (384) | 8GB 安全值 |

### 架构讨论：手机端部署方案

- **结论**：当前架构无需修改，天然支持手机端
- **原理**：本项目采用「服务器端渲染 + WebRTC 推流」架构
  - GPU 服务器负责全部推理（Wav2Lip + TTS）
  - 手机端只需集成 WebRTC 播放器接收视频流
  - 无需在手机上部署任何 Python/PyTorch/模型文件
- **方案对比**：
  - 方案 A（推荐）：服务器推理 + 手机 WebRTC 播放 —— 任何智能手机均可，1-2 周开发
  - 方案 B（不推荐）：手机本地推理 —— 仅旗舰机可行，需模型转换（ONNX），2-3 月开发
- **手机端接入方式**：`http://服务器IP:8010/offer` WebRTC 握手 → 接收 H.264 视频流
- **参考**：与 HeyGen、D-ID、硅基智能等商用产品架构一致

## 2026-05-28（周三）— 分辨率全链路深度分析

### 背景

与 AI 探讨了"输出视频分辨率到底是多少"的问题，澄清了 full_imgs / face_imgs / 模型内部三层分辨率的区别。

### 三条核心结论

#### 结论 1：full_imgs 分辨率由 `max_side=1280` 决定

`genavatar.py` → `video2imgs()` 第 20 行：

```python
if max(h, w) > max_side:   # max_side=1280
    scale = max_side / max(h, w)
    frame = cv2.resize(frame, ...)
```

逻辑是 `max(宽, 高) > 1280` **才**触发缩放：

| 视频 | 原始分辨率 | max(宽,高) | >1280? | full_imgs |
|------|-----------|------------|--------|-----------|
| 4K 视频 | 3840×2160 | 3840 | ✅ | 1280×720 |
| 1080p | 1920×1080 | 1920 | ✅ | 1280×720 |
| **static.mp4** | 1280×1280 | 1280 | ❌ 等于，绕过 | **1280×1280** |
| xp | 404×720 | 720 | ❌ | 404×720 |

> static.mp4 恰好正方形 1280×1280，`max=1280` 不满足 `>` 条件，完美绕过缩放，所以是 1280×1280 而非 720p。

#### 结论 2：face_imgs 永远是 256×256（核心瓶颈）

不管输入是 4K 还是 720p，`genavatar.py` 都会做：

```python
resized_crop_frame = cv2.resize(face_frame, (img_size, img_size))  # 256×256
```

**举例（半身人像 1080×1920）：**

```
人脸区域在原图中 ≈ 600×700 像素
  → cv2.resize → 256×256  喂给模型
  → 模型推理 → 输出 256×256
  → cv2.resize → 600×700  贴回 full_imgs
```

模型**从未见过** 600×700 的原始人脸细节。256×256 里没有的信息，拉伸到 600×700 也不会有。就像低分辨率图片放大——不会凭空变出细节。

**三层分辨率总结：**

| 层级 | 分辨率 | 决定因素 |
|------|--------|----------|
| full_imgs（整帧） | 最高 1280px 宽边 | `video2imgs` 的 `max_side=1280` |
| face_imgs（人脸） | 固定 256×256 | `genavatar` 的 `--img_size 256` |
| 模型内部 | 256×256 或 192×192 | 模型架构（256/384） |

**所以即使 full_imgs 是 4K，人脸区域永远是 256×256 的画质。** 背景、衣服、头发不走模型，是高清的；但人脸（尤其嘴唇）的上限就是 256×256。

#### 结论 3：输出帧率固定 25 FPS

`server/webrtc.py` 第 30 行：

```python
VIDEO_PTIME = 0.040  # 1/25 → 25 FPS
```

推流时钟固定 25fps，模型推理按音频 mel 帧节奏走，WebRTC 按 25fps 打时间戳。

### 对清晰度优化的启示

提高画质有两个方向：
1. **提高 `max_side`**（如改到 1920）：full_imgs 保留 1080p，但 x264 编码器可能崩溃（syl 就是 1080p 黑屏），需配合 VP8 编码器
2. **换 384 模型**：不蠢不蠢 384 用 SAM 空间注意力，人脸细节理论上更好（待测试）
